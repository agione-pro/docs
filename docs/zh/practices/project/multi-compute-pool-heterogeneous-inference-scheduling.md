# 第一章 背景与架构总览

## 1.1 业务背景

本文档面向已将自微调 Qwen-32B 模型投入生产的企业，描述在异构硬件（华为昇腾 910B NPU + 英伟达 A10 GPU）、跨网络域（华为公有云 HCS、本地 IDC 私有云、IDC 物理服务器）的条件下，如何通过 IMMS AGIOne 平台实现多算力池统一管理、弹性调度与高可用运营。

## 1.2 核心挑战

* 硬件异构：Ascend 910B 与 NVIDIA A10 两类 GPU/NPU 并存，推理引擎（MindIE / vLLM）各异；
* 节点离散：算力分布于华为 HCS 公有云、本地 IDC 私有云及 IDC 裸金属三类环境，网络隔离；
* Ascend 节点故障时操作系统自动重装，需自动化恢复，保障推理服务连续性；
* 实例规模从 4 个扩展至 20-36 个，横向扩缩不影响上游调用；
* 多时段差异化负载（夜间批量报告、白天小流量、白天批量报告），需按场景分配资源；
* 推理任务耗时不均，简单轮询负载均衡导致实例负载失衡；
* 调用异常时，需联动查看吞吐量与 NPU/GPU 实例负载，快速定位根因。

## 1.3 总体架构

IMMS AGIOne 平台采用「中心管控 + 边缘执行」架构，分为管控平面、调度平面与执行平面三层。

| **架构层次** | **核心职责与组件** |
|---|---|
| 管控平面（HCS 私有云） | AGIONE 管理服务：API 网关、核心调度器、模型发布、计费监控。统一对外暴露模型 API，管理所有算力集群的注册与心跳。 |
| 调度平面（各算力集群） | 每个算力集群（HCS NVIDIA A10、ModelArts Lite Ascend、IDC Ascend 物理机）独立部署 K8s 基础服务、调度组件、监控组件，并对管控平面开放任务构建 API。 |
| 执行平面（推理实例） | MindIE 容器（Ascend 910B）或 vLLM 容器（NVIDIA A10），加载 Qwen-32B 模型权重，提供 OpenAI 兼容推理接口。 |
| 聚合调度层 | 聚合模型（Aggregated Model）按场景（全量 / 小部分 / 大部分）封装多个推理实例，提供动态加权负载均衡；RPM、TPM、TTFT、Running Tasks 等指标实时采集并驱动分发策略自适应调整。 |

> 管控平面部署于 HCS 私有云，通过专线连接 ModelArts Lite 节点，通过内部路由连接 IDC 物理机。NVIDIA A10 集群通过 HCS VPC 接入。所有节点统一纳入 AGIONE 多集群视图。

## 1.4 网络连接矩阵

| **节点类型** | **与 AGIONE 管控的连接方式** | **备注** |
|---|---|---|
| HCS NVIDIA A10 服务器 | HCS VPC 内网直连 | 延迟最低，推荐显存低，部署 Qwen3.5-9B 等小模型，用于内部简单问答。 |
| ModelArts Lite Ascend 910B | 专线网络连接 | 稳定独占带宽，适合批量推理 |
| IDC Ascend 910B 物理服务器 | 内部交换机 / 路由器连接 | 裸金属性能强，推荐夜间全量批处理 |

# 第二章 多算力池接入与管理

## 2.1 集群纳管策略

AGIONE 管理服务采用「多池多集群」模型，将三类算力环境统一纳管，每个集群保持独立的 K8s 控制平面，仅通过标准化任务构建 API 与 AGIONE 交互。

### 2.1.1 NVIDIA A10 集群接入（HCS VPC 模式）

1. 在 HCS VPC 内创建 K8s 集群，部署 GPU 调度插件（nvidia-device-plugin）。
2. 在集群内安装 AGIONE 调度组件、监控组件等。
3. AGIONE 调度组件向管控平面注册集群信息（集群 ID、GPU 型号、可用 GPU 数、网络端点）。
4. 管控平面下发心跳探测任务（每 60 秒），持续监控集群健康状态。

### 2.1.2 ModelArts Lite Ascend 910B 接入（专线模式）

1. 在 ModelArts Lite Server 云主机上部署 K8s 基础服务，安装 Ascend Device plugin 昇腾设备调度插件。
2. 部署 AGIONE 调度组件，配置专线网络端点与 AGIONE 管控通信。
3. 配置 NPU 监控组件（npu-exporter），采集 NPU 利用率、显存、温度等指标。
4. 注册集群至 AGIONE，标注节点属性（硬件类型: ascend-910b64g）。

### 2.1.3 IDC Ascend 910B 物理服务器接入（内网路由模式）

1. 物理服务器安装操作系统后，执行 AGIONE 提供的节点初始化脚本（安装 Docker、K8s kubelet、昇腾驱动）。
2. 通过内部交换机与路由器配置与 AGIONE 管控的路由可达。
3. 部署 AGIONE 调度组件与监控组件，完成集群注册。
4. 将重要配置数据与持久化模型缓存挂载至数据盘（非系统盘）。

## 2.2 集群资源标签与调度

AGIONE 通过节点标签（Labels）实现调度控制，确保不同聚合模型的请求路由到合适的算力池。

| **标签键** | **示例值** | **含义** | **推荐调度场景** |
|---|---|---|---|
| hardware-type | ascend-910b64g / nvidia-a1024g | 硬件类型 | 按推理引擎路由 |

# 第三章 Ascend 节点故障自愈机制

## 3.1 故障触发与恢复流程

昇腾 910B 节点发生硬件异常时，华为云节点故障触发机制将自动执行操作系统重装（Re-image）。AGIONE 平台围绕此机制，构建了端到端的快速自恢复流程。

> **自恢复流程（端到端目标）：** 节点故障告警 → 自动摘除流量 → OS 重装（约 10min）→ 基础服务快速恢复（<5min）→ 推理服务启动（约 <15min）→ 健康检查通过 → 自动接入流量

## 3.2 基础镜像制作规范

节点初始化完成后，应立即制作基础镜像（Base Image），将操作系统配置、驱动、K8s 组件、AGIONE Agent 等封固在镜像中，缩短恢复时间。

| **镜像层次** | **包含内容** |
|---|---|
| OS 基础层 | EulerOS / Ubuntu 22.04 LTS + 安全补丁 + 时区 / DNS 配置 |
| 昇腾驱动层 | CANN 工具链、昇腾驱动、固件版本锁定 |
| 容器运行时层 | Docker CE、containerd、K8s kubelet（版本锁定）、nvidia/ascend device plugin |
| AGIONE 组件层 | 监控组件、日志采集器、节点注册脚本（自动注册至管控平面） |
| 推理引擎层（可选） | MindIE 基础镜像 / vLLM 基础镜像（含 Python 环境、依赖包，模型权重通过数据盘挂载） |

## 3.3 重要数据存储规范

* 模型权重文件：存储于数据盘（独立于系统盘），如挂载路径 `/data/models/<model-name>/`。OS 重装不影响数据盘。
* K8s 集群配置（kubeconfig、etcd 快照）：存储于数据盘映射文件夹。

## 3.4 流量摘除与自动恢复联动

1. AGIONE 心跳探测失败 2 次（60 秒）后，将节点标记为 NotReady，聚合调度层自动将该节点的权重设为 0，停止新请求路由至该节点。
2. 节点重装完成后，自动接入到集群。
3. 管控平面验证注册信息后，触发健康检查探针（HTTP /health 接口），连续 3 次成功后将节点权重恢复至默认值。
4. 聚合调度层恢复向该节点路由请求，运营监控面板自动更新节点状态。

# 第四章 推理引擎部署与性能基线

## 4.1 MindIE 推理引擎（昇腾 910B）

### 4.1.1 基础镜像构建

* 基础镜像：基于 CANN 官方镜像（mindspore/mindie:latest-ascend910b）定制。
* 安装 Qwen-32B 所需的 MindFormers 推理框架，锁定版本防止依赖漂移。
* 预加载分词器（Tokenizer）至镜像，避免首次启动时网络下载延迟。
* 镜像中嵌入启动脚本，支持通过环境变量注入模型路径、端口、并发参数。

### 4.1.2 关键参数优化

| **参数** | **推荐值 / 说明** |
|---|---|
| max-seq-len | 根据业务最大上下文长度设置，不超过显存上限 |
| max-iter-times | 输出 token 量 |
| max-input-token-len | 输出 token 量 |
| world-size | Ascend 910B 单机 8 卡可设为 8；跨机张量并行需配置 HCCL 通信 |
| block-size | 16（PagedAttention KV Cache 块大小，建议 16 或 32） |
| max-batch-size | 0.90（预留 10% 显存余量，防止 OOM） |

## 4.2 vLLM 推理引擎（NVIDIA A10）

### 4.2.1 基础镜像构建

* 基础镜像：基于 vllm-project/vllm-openai 官方镜像，添加 Qwen-32B 适配补丁。
* 启用 FlashAttention-2 加速内核（A10 支持 BF16/FP16 混合精度）。
* 配置 CUDA 环境变量，启用 NCCL 多机通信（如实例跨机分布时使用）。

### 4.2.2 关键参数优化

| **参数** | **推荐值 / 说明** |
|---|---|
| dtype | bfloat16（A10 支持 BF16，精度与速度平衡优于 fp16） |
| max-model-len | 根据业务最大上下文长度设置，不超过显存上限 |
| tensor-parallel-size | 并行运行，qwen3.5-9b 设置为 2 |
| gpu-memory-utilization | 0.90 |
| max-num-seqs | 256（最大并发序列数，根据 GPU 显存调整） |
| quantization | awq 或 gptq（推荐 AWQ INT4，显存减半，精度损失可控） |

## 4.3 性能压测方法（evalscope）

使用 evalscope 工具模拟上游业务请求，评估单实例性能基线。压测时应尽量模拟真实业务上下文长度分布，而非固定短文本。

### 4.3.1 压测场景配置

| **压测场景** | **Prompt Token 长度** | **最大 Output Token** |
|---|---|---|
| 短文本 | < 1k | < 1k |
| 中等文本 | < 10k | 5k |
| 长文本 | < 32k | 10k |

### 4.3.2 关键性能指标目标参考

| **指标** | **含义** | **Ascend 910B 参考值** | **NVIDIA A10 参考值** |
|---|---|---|---|
| TTFT（首 Token 时延） | 从提交请求到收到第一个 Token 的时间 | < 2.5s（FP16） | < 1.8s（AWQ INT4） |
| TPS（Token/s） | 每秒生成 Token 数（单实例） | ~180 t/s（FP16 TP8） | ~120 t/s（INT4 TP4） |
| RPM（请求/分钟） | 单实例每分钟处理请求数 | ~20 ~ 40 RPM | ~30 ~ 60 RPM |
| TPM（Token/分钟） | 单实例每分钟处理 Token 数 | ~8,000 ~ 15,000 | ~6,000 ~ 12,000 |
| GPU 利用率 | 推理负载下的 NPU/GPU 计算利用率 | > 70%（健康态） | > 65%（健康态） |

# 第五章 聚合模型与动态调度策略

## 5.1 聚合模型设计

聚合模型（Aggregated Model）是 AGIONE 平台的核心抽象：它在逻辑上代表一个统一的模型，在物理上由多个后端推理实例组成。上游调用者无需感知后端实例的分布，仅与聚合模型交互，与普通模型一样调用。

## 5.2 三类业务场景聚合模型配置

### 5.2.1 小部分负载聚合模型（白天 API 支撑）

| **配置项** | **推荐值 / 说明** |
|---|---|
| 聚合模型 ID | 对外暴露独立模型 ID |
| 后端实例数量 | 3 ~ 5 个 |
| 负载均衡策略 | 体验+轮询均衡（依据 Running Tasks 数量实时调整） |
| 超时配置 | 单请求超时 3000s（针对长输入、输出请求） |
| QPS 上限 | 建议设置限流（如 20 QPS），防止突发流量压垮少量实例 |
| 适用时段 | 工作日，如 08:00 ~ 20:00 白天业务调用 |

### 5.2.2 大部分常规负载聚合模型（白天批量报告）

| **配置项** | **推荐值 / 说明** |
|---|---|
| 聚合模型 ID | 对外暴露独立模型 ID |
| 后端实例数量 | 10 ~ 30 个 |
| 负载均衡策略 | 体验+轮询均衡（依据 Running Tasks 数量实时调整） |
| 超时配置 | 单请求超时 3000s（长文本生成） |
| 并发控制 | 每实例最大 Running Tasks 配置不超过压测基线的 99%，否则均衡分配进入排队。 |
| 适用时段 | 工作日 09:00 ~ 20:00 批量报告生成 |

### 5.2.3 全量负载聚合模型（夜间全量批处理）

| **配置项** | **推荐值 / 说明** |
|---|---|
| 聚合模型 ID | 对外暴露独立模型 ID |
| 后端实例数量 | 全部实例 |
| 负载均衡策略 | 体验+轮询均衡（依据 Running Tasks 数量实时调整） |
| 超时配置 | 单请求超时 3000s（长文本生成） |
| 批处理优化 | 启用请求合批（Batching）模式，提升 TPS |
| 适用时段 | 每日 20:00 ~ 次日 08:00 夜间批量任务 |

## 5.3 动态分发调度机制

AGIONE 动态调度器每 1 分钟统计一次各后端实例的实时指标，并据此更新路由权重，解决因推理任务耗时不均导致的实例负载失衡问题。

> **权重计算公式（简化版）：**
>
> *Weight(i) = BaseCapacity(i) × HealthScore(i) / (RunningTasks(i) + 1)*
>
> * **BaseCapacity**：实例压测 TPM 基线归一化值（体现硬件性能差异）
> * **HealthScore**：0.0 ~ 1.0（Success Rate），异常时降为 0
> * **RunningTasks**：当前正在处理的请求数（实时采集）

# 第六章 横向扩缩容最佳实践

## 6.1 扩容操作流程（不影响在线业务）

1. 在目标算力集群（任意一类）准备新节点，执行基础镜像初始化。
2. 新节点启动后自动向管控平面注册，初始权重为 0（不接收流量）。
3. 在 AGIONE 管理控制台创建新推理实例，关联到目标聚合模型（如仅加入「全量负载」聚合模型，不影响「小部分负载」端点）。
4. 推理服务健康检查通过后，将实例加入聚合模型的后端列表，权重从 0 线性增长至基线（预热时间约 5 分钟，防止冷启动冲击）。
5. 在运营监控面板确认新实例 RPM、TTFT 指标正常后，扩容完成。

> 扩容期间聚合模型端点 URL 不变，上游调用者无感知。建议在业务低谷期（如夜间）执行扩容操作，以便充分验证。

## 6.2 缩容操作流程（优雅下线）

1. 在 AGIONE 控制台将目标实例标记为「禁用」状态，调度器停止向该实例路由新请求。
2. 等待该实例 Running Tasks 降至 0（最长等待时间 = 请求超时 × 2）。
3. 实例 Running Tasks 为 0 后，从聚合模型后端列表中摘除，销毁推理容器。
4. 若实例所在节点也需下线，待该节点上运行的所有模型实例无任务运行后，执行自动化脚本，触发 K8s 执行节点排水（kubectl drain）。

# 第七章 运营监控与可观测性

## 7.1 监控指标体系

AGIONE 运营监控体系分为三层：应用层（调用指标）、调度层（聚合模型指标）、基础设施层（GPU/NPU 资源指标），三层指标联动，支持快速定位异常根因。

### 7.1.1 应用层指标（用户视角）

| **指标名称** | **说明与告警阈值建议** |
|---|---|
| RPM（Requests Per Minute） | 每分钟请求数 |
| RPH（Requests Per Hour） | 每小时请求数 |
| RPD（Requests Per Date） | 每天请求数 |
| TPM（Tokens Per Minute） | 每分钟处理 Token 数，反映整体吞吐 |
| TPH（Tokens Per Hour） | 每小时处理 Token 数，反映整体吞吐 |
| TPD（Tokens Per Day） | 每天处理 Token 数，反映整体吞吐 |
| TTFT（Time To First Token） | 首 Token 时延 |
| P50/P95/P99 端到端时延 | 请求全链路时延分布 |
| Success Rate | 请求成功率 |
| Error Rate by Type | 按错误码分类统计，区分客户端错误、平台服务端、模型实例处错误 |

### 7.1.2 调度层指标（聚合模型视角）

| **指标名称** | **说明** |
|---|---|
| Running Tasks（各实例） | 各后端实例当前正在处理的请求数；是动态调度权重的核心输入 |
| 实例权重分布 | 各实例当前权重可视化；用于评估负载均衡效果 |
| 路由分发比例 | 各实例实际接收的请求占比；与权重对比，验证调度策略生效 |
| 熔断/降级次数 | 调度层触发熔断的实例数及频次 |

### 7.1.3 基础设施层指标（实例资源视角）

| **指标名称** | **说明与告警阈值建议** |
|---|---|
| NPU/GPU 计算利用率 | < 20% 提示过度空闲（浪费），> 90% 持续 5 分钟触发扩容告警 |
| NPU/GPU 显存使用率 | > 90% 触发告警（OOM 风险） |
| NPU/GPU 温度 | > 85°C 触发告警，> 90°C 触发紧急通知 |
| CPU 利用率 | 推理服务通常 CPU 占用低；> 80% 可能提示数据预处理瓶颈 |
| 内存使用率 | > 85% 触发告警 |
| 节点健康状态 | K8s Node Condition（Ready/NotReady）；NotReady > 60s 触发自愈流程 |

## 7.2 异常联动排查流程

当用户反馈调用异常（报错或响应变慢）时，建议按以下流程逐层排查。

> **异常排查流程（推荐顺序）：**
>
> **Step 1** 查看聚合模型 RPM、Success Rate 趋势图，定位异常时间窗口。
>
> **Step 2** 查看各后端实例 Running Tasks 分布，识别负载失衡实例（Running Tasks 异常高或持续不降）。
>
> **Step 3** 对负载异常实例，同步查看其 NPU/GPU 利用率、显存、温度；高利用率 + 高 Running Tasks = 性能瓶颈，低利用率 + 高 Running Tasks = 请求堆积（可能 OOM 或死锁）。
>
> **Step 4** 查看推理服务日志，搜索 CUDA/NPU OOM、timeout、connection refused 等关键字。
>
> **Step 5** 确认根因后：① 重启故障实例（Running Tasks 归 0 后）；② 调整调度策略临时绕开该实例；③ 提交节点告警触发自愈流程。

## 7.3 运营分析与扩缩容决策

AGIONE 运营分析面板提供以下能力，为扩缩容决策提供数据支撑。

* 吞吐量趋势分析：按时段（小时/天/周）展示 RPM/RPH/RPD、TPM/TPH/TPD 趋势，识别业务高峰规律，提前规划资源。
* 资源利用率热力图：按实例 × 时段展示 NPU/GPU 利用率，识别长期低利用率实例（缩容候选）。
* TTFT 分布分析：P50/P95/P99 时延分布，评估用户体验；P95 持续超标时应优先扩容。
* 成本效益分析：基于 TPM 与资源消耗计算单 Token 成本，对比不同硬件类型的性价比，指导采购决策。
* 容量预测：基于历史趋势外推未来 30 天 RPM/TPM，提前 7 天告警资源不足风险。

# 第八章 安全与合规

## 8.1 网络安全

* 所有跨网络域通信（HCS ↔ 专线 ↔ IDC）均需启用 TLS 1.2+ 加密；
* AGIONE API 网关启用身份认证（API Key / OAuth 2.0），按调用方分配独立密钥。
* 推理实例端口不对外暴露，仅通过 AGIONE 聚合模型端点代理访问。
* 定期审计 VPC 安全组规则，清理过期的开放端口。

## 8.2 数据安全

* 推理请求中若包含敏感用户数据，在 API 网关层启用数据脱敏或审计日志。
* 日志中禁止记录推理请求的 Prompt 与输出内容（仅记录 Token 数、请求 ID、耗时）。
