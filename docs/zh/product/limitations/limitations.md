# AGIOne 产品限制条件

## 1. 文档说明

本文档用于说明 AGIOne 当前产品支持范围中的限制条件、风险项和待确认事项。

## 2. 总体限制说明

| 范围 | 限制条件 |
| --- | --- |
| 管理节点部署 |具体规格需按部署形态、纳管规模和交付包版本确认 |
| 算力节点纳管 | 芯片可纳管不等于所有模型、驱动、CUDA、推理引擎组合均已验证 |
| 算力平台纳管 | 不同云平台的账号权限、网络互通、API 密钥和安全策略需单独评估 |
| 模型服务能力 |具体能力依赖模型本身、推理引擎、协议适配和产品版本 |
| 快速部署模型 | 超大模型需评估显存、并行策略、量化方式、存储和网络条件 |

## 3. 管理节点部署限制条件

| 部署形态 | 限制条件 |
| --- | --- |
| All-in-One 单机部署 |适合演示、测试、PoC 或小规模部署，不建议承载高可用生产负载 |
| 本地 Host-mode 多节点部署 | 历史数据、端口占用、目录权限和数据库主备风险需部署前确认 |
| Kubernetes / 云原生部署 | Kubernetes版本需按交付包兼容矩阵确认；存储、CNI 和 Ingress 差异会影响交付 |
| 私有化部署 | 网络隔离、离线包、证书、DNS、代理、安全基线和审计策略需现场评估 |
| 离线或弱网部署 | 缺失离线资源、镜像导入失败、仓库不可达会阻塞部署 |

## 4. 管理节点操作系统与组件限制条件

| 类别 | 限制条件 |
| --- | --- |
| 操作系统 |生产应以正式 release note、交付包兼容矩阵和现场验证结果为准 |
| Docker |需匹配操作系统、内核、镜像包和离线安装方式 |
| Kubernetes |需结合交付包、GPU/NPU 插件、CNI、CSI、Ingress Controller 验证 |
| Helm |仅适用于 Helm 或 chart交付形态 |
| 中间件 |端口冲突、存储持久化、备份恢复和安全基线需部署前确认 |

## 5. 纳管算力节点限制条件

| 厂商 | 架构 / 系列 | 限制条件 |
| --- | --- | --- |
| NVIDIA | Hopper |需结合驱动、CUDA、推理引擎、显存和网络验证 |
| NVIDIA | Ampere | 不同型号显存差异大，不能直接等同支持所有模型规格 |
| NVIDIA | Ada | 消费级或工作站型号需额外确认驱动、散热、稳定性和数据中心交付要求 |
| Huawei Ascend | Ascend910 |需结合 CANN、MindIE、模型适配和镜像版本确认 |
| Enflame | Enflame |需结合驱动、推理框架和模型适配验证 |
| Biren | Biren |需结合驱动、推理框架和模型适配验证 |

## 6. 算力节点操作系统限制条件

| 节点类型 | 限制条件 |
| --- | --- |
| NVIDIA GPU 算力节点 |具体 OS版本需结合驱动和交付包确认 |
| Ascend NPU 算力节点 |需按 Ascend 软件栈版本和模型适配情况确认 |
| 国产加速卡算力节点 | 不同国产芯片生态差异较大，需项目级验证 |
| Kubernetes Worker 节点 | 同一 Kubernetes 集群建议管理相同 CPU 架构，避免异构架构混入导致调度不稳定 |

## 7. CUDA 限制条件

| 项目 | 限制条件 |
| --- | --- |
| CUDA 固定版本声明 | CUDA需结合 NVIDIA GPU 型号、驱动版本、容器镜像、推理引擎和 AGIOne交付包兼容矩阵确认 |
| NVIDIA GPU 推理 | 不同模型对 CUDA、显存、TensorRT-LLM、vLLM、SGLang 等依赖不同 |
| CUDA 与模型部署 | 支持某 GPU 型号不等于支持任意 CUDA版本或任意模型镜像 |
| CUDA 与生产交付 | 不建议在生产中混用未验证 CUDA / Driver / Runtime组合 |

## 8. 纳管算力平台限制条件

| 平台类型 | 限制条件 |
| --- | --- |
| 阿里云 |需确认账号权限、网络互通、密钥管理 |
| AWS |需确认 Region、IAM 权限、API 密钥 |
| Google Cloud |需确认项目权限、API 启用 |
| AGIOne 异构卡纳管（算模方） |需确认平台接口和接入方式 |
| 华为云 | 当前暂不支持，未来版本规划中 |

## 9. 模型能力限制条件

| 能力类型 | 限制条件 |
| --- | --- |
| 文本生成 / Chat |受模型上下文长度、显存、吞吐和推理引擎影响 |
| Reasoning / 思考模型 | 推理模型通常有更高时延和 token 消耗，需要容量评估 |
| 多模态 / Vision-Language |依赖模型镜像、视觉预处理、显存和推理框架支持 |
| OCR | OCR 效果和吞吐依赖图片质量、模型版本和推理资源 |
| Embedding | RAG 场景需结合向量库、chunking、权限过滤和召回策略 |
| Reranker | 增加召回质量的同时会增加请求链路时延 |
| RAG |需治理知识权限、数据切分、向量化、召回和审计 |
| Function Calling / Tool Use |需确认模型协议、工具 schema、权限和执行审计 |
| Agent / Workflow |依赖 Agent Builder、工具集成和企业权限体系 |
| OpenAI-compatible API |具体接口范围需按产品版本确认 |
| Anthropic-compatible API |需按产品版本、上游模型和协议策略确认 |
| SSE 流式输出 | 不同上游 provider 的 usage 字段和事件格式可能存在差异 |
| 聚合模型路由 | 不同模型能力、上下文长度、工具调用能力不一致时需要策略约束 |
|计量计费 |计费规则、License 和 Credential 抽象需按商业规则确认 |

## 10. 部署与运行限制条件

| 限制项 |说明 | 建议 |
| --- | --- | --- |
| 管理节点与算力节点角色不同 | 管理节点不要求 GPU/XPU；算力节点至少需要1 个 XPU/GPU/NPU | 部署设计中明确管理面和算力面边界 |
|资源规格存在多口径 | All-in-One 推荐8C/16GiB/200GiB；生产管理节点要求可达 >=16C/>=16GiB，至少3 节点 | 按部署形态区分 all-in-one、host-mode 多节点、生产管理面 |
|端口入口存在多口径 | 文档中存在 `443`、`18090`、`80`、`8089` 等入口 | 区分外部 HTTPS入口、门户/API入口、Nginx入口和作业访问代理入口 |
| 离线交付依赖完整包 | 缺少 bundle、镜像、离线 Python、checksum 或 Docker/Compose资源会阻塞部署 | 部署前执行完整性交付物校验 |
| 同构架构优先 | 同一 Kubernetes 集群建议管理相同 CPU 架构 | 异构 CPU 架构建议拆分集群或资源池 |
| 时间同步要求 | 多节点部署要求 NTP 或等效时间同步 | 时钟偏差建议控制在1 秒以内 |
|生产资源冗余 | 高并发和核心业务需要预留资源头寸 |生产建议预留20%-30%资源冗余 |

## 11. 模型部署限制条件

| 限制项 |说明 | 建议 |
| --- | --- | --- |
| 芯片兼容不等于模型验证 | 支持某芯片纳管，不代表该芯片已验证所有模型、推理引擎和量化方案 | 建立模型-芯片-驱动-推理引擎验证矩阵 |
| 超大模型资源门槛高 | 如 qwen3.5-397b、qwen3-235b、DeepSeek-V3 等需要高显存、多卡或并行策略 | 部署前评估显存、卡数、并行、KV Cache、上下文长度和吞吐 |
| 多模态和 OCR依赖额外链路 |视觉模型和 OCR 模型依赖图像预处理、模型镜像和推理框架支持 | 按模型类型单独验证端到端链路 |
| Embedding / Reranker影响 RAG 延迟 | 增加向量化和重排序后会增加链路耗时 | 对 RAG 场景单独压测召回质量、时延和并发 |
| Reasoning 模型消耗更高 | 思考模型通常输出更长、耗时更高、token 消耗更大 | 单独设置限流、额度和成本策略 |
| 聚合模型能力需对齐 | 聚合模型下不同子模型的上下文、工具调用、多模态、价格和协议能力可能不同 | 路由策略中增加能力约束和降级策略 |

## 12. 网络、存储与高可用限制条件

| 限制项 |说明 | 建议 |
| --- | --- | --- |
| RDMA 要求按场景区分 | 单卡推理可不强依赖 RDMA；多机多卡推理或训练建议100Gbps以上 RDMA | 按模型规模、并行策略和吞吐目标设计网络 |
|共享存储依赖模型规模 | 模型权重、镜像、日志、MinIO、数据库和中间件需要持久化存储 | 提前规划块存储、NAS、NFS、Ceph 或 RWX 存储 |
| 云原生部署依赖 StorageClass | 数据库和中间件持久卷需要 RWO，部分共享数据需要 RWX | 部署前验证 PVC 创建、挂载和备份恢复 |
| 镜像仓库必须可达 | 云原生部署要求所有节点可访问镜像仓库 | 提前验证 registry、认证 Secret 和 image pull |
| Ingress / DNS / TLS影响访问 | 云原生入口依赖域名解析、证书和入口控制器 | 上线前完成外部访问、内部解析和证书验证 |
| 高可用需要多节点设计 | 单机部署无法提供完整高可用 |生产建议采用多管理节点、数据库主备、备份和回滚机制 |

## 13. 待确认事项

| 待确认项 | 当前状态 | 建议动作 |
| --- | --- | --- |
| CUDA版本矩阵 | 当前知识库未声明固定 CUDA版本 | 根据交付包、GPU驱动、镜像和推理引擎补充 CUDA 矩阵 |
| GPU 驱动版本矩阵 | 当前只明确芯片型号，未给出驱动版本 | 建立 GPU型号-Driver-CUDA-推理引擎-镜像版本矩阵 |
| Ascend 软件栈版本 | 当前只明确 Ascend910B/910C 支持 | 补充 CANN、MindIE、驱动和模型适配版本 |
| 模型显存与卡数要求 | 当前清单只列模型名称 | 按模型规模补充 FP16/BF16/INT8/INT4、张量并行、最小卡数 |
| API 协议支持范围 | 白皮书提到 OpenAI / Anthropic 等协议 | 按正式产品版本确认接口范围和字段兼容性 |
|端口口径 | 存在 `443`、`18090`、`80`、`8089` 等多种表达 |形成正式网络端口规划表 |
